Build Only When Relevant Files Changed
push hard perf
Likely prompt
"Run a frontend build only when files under src/ or package.json changed in the push. If only docs or config changed, skip the build entirely to save resources."
No native "changed files" signal field — use git diff after checkout
Compare pushed SHA against its parent (HEAD^)
Write skip flag to OUTPUT_PATH, use in validation expression
This saves CPU/memory bonus points on the leaderboard
push → checkout → git diff HEAD^ → grep src/ package.json ├─ match → build → validation=true └─ no match → skip → validation=false
Answer
signals:
  - push

vars:
  REPO_URL: "https://@<< secrets.GH_TOKEN >>@@<< signal.source >>.com/@<< signal.owner >>/@<< signal.repo >>"
  PUSH_SHA: "@<< signal.push.sha >>"

steps:
  - id: checkout
    exec: |
      set -e
      git clone "$REPO_URL" .
      git checkout "$PUSH_SHA"

  - id: check-changes
    exec: |
      # Get files changed in this commit vs parent
      CHANGED=$(git diff --name-only HEAD^ HEAD 2>/dev/null || \
                git diff --name-only HEAD 2>/dev/null)
      echo "Changed files:"
      echo "$CHANGED"

      if echo "$CHANGED" | grep -qE '^(src/|package\.json|package-lock\.json)'; then
        echo "Relevant files changed — building."
        echo "should_build=true" >> "$OUTPUT_PATH"
      else
        echo "No relevant changes — skipping build."
        echo "should_build=false" >> "$OUTPUT_PATH"
      fi

  - id: build
    exec: |
      if [ "$SHOULD_BUILD" != "true" ]; then exit 0; fi
      set -e
      npm ci && npm run build
    vars:
      SHOULD_BUILD: "@<< steps.check-changes.should_build >>"

validation: "@<< steps.check-changes.should_build >>"
Tip: git diff HEAD^ HEAD fails on the very first commit (no parent). The || git diff --name-only HEAD fallback handles that case.

push send expert

"Build the project for three targets (linux/amd64, linux/arm64, darwin/amd64) in parallel. Use separate processors triggered by sends from a dispatcher. Collect all three artifacts."

push → dispatcher ├─ send → build-target {os:linux, arch:amd64} ├─ send → build-target {os:linux, arch:arm64} └─ send → build-target {os:darwin, arch:amd64}
Processor 1: dispatcher (push)
signals:
  - push

vars:
  SHA: "@<< signal.push.sha >>"

steps:
  - id: trigger-matrix
    exec: echo "Dispatching matrix builds for $SHA"

sends:
  - name: build-target
    inputs:
      sha:  "@<< signal.push.sha >>"
      os:   linux
      arch: amd64
  - name: build-target
    inputs:
      sha:  "@<< signal.push.sha >>"
      os:   linux
      arch: arm64
  - name: build-target
    inputs:
      sha:  "@<< signal.push.sha >>"
      os:   darwin
      arch: amd64
Processor 2: build-target (send)
signals:
  - send

validation: true

vars:
  SHA:     "@<< signal.send.inputs.sha >>"
  GOOS:    "@<< signal.send.inputs.os >>"
  GOARCH:  "@<< signal.send.inputs.arch >>"

Auto-tag Semantic Version on Push to Main
push hard
Likely prompt
"When a push lands on main, read the current version from package.json (or VERSION file), auto-increment the patch number, commit the new version file, and push a git tag v<major>.<minor>.<patch>."
Filter to refs/heads/main only
Use jq to parse package.json version
Increment patch with shell arithmetic
Configure git identity before commit/push
Guard against infinite loop: check if last commit is already a version bump
Answer
signals:
  - push

validation: true

vars:
  REF:      "@<< signal.push.ref >>"
  PUSH_SHA: "@<< signal.push.sha >>"
  REPO_URL: "https://@<< secrets.GH_TOKEN >>@@<< signal.source >>.com/@<< signal.owner >>/@<< signal.repo >>"

steps:
  - id: checkout
    exec: |
      set -e
      [ "$REF" != "refs/heads/main" ] && exit 0
      git clone "$REPO_URL" .
      git fetch origin "$PUSH_SHA" && git checkout "$PUSH_SHA"

  - id: bump-version
    exec: |
      set -e
      # Guard: skip if this commit is already a version bump
      LAST_MSG=$(git log -1 --pretty=%s)
      if echo "$LAST_MSG" | grep -q "^chore: bump version"; then
        echo "Already a version bump commit. Skipping."
        exit 0
      fi

      # Read + bump version from package.json
      CURRENT=$(jq -r .version package.json)
      MAJOR=$(echo "$CURRENT" | cut -d. -f1)
      MINOR=$(echo "$CURRENT" | cut -d. -f2)
      PATCH=$(echo "$CURRENT" | cut -d. -f3)
      NEW_PATCH=$((PATCH + 1))
      NEW_VERSION="$MAJOR.$MINOR.$NEW_PATCH"

      # Write back to package.json
      jq --arg v "$NEW_VERSION" '.version = $v' package.json > tmp.json
      mv tmp.json package.json

      # Commit and tag
      git config user.email "ci@pulse-runner"
      git config user.name "Pulse Runner CI"
      git add package.json
      git commit -m "chore: bump version to $NEW_VERSION"
      git tag "v$NEW_VERSION"
      git push origin main --tags
      echo "new_version=$NEW_VERSION" >> "$OUTPUT_PATH"
Watch out: The push from this step will itself fire a push signal. The guard on the commit message (chore: bump version) prevents an infinite loop.

Cross-repo Trigger (push to A → act on B)
push hard new pattern
Likely prompt
"When repo 'backend' gets a push to main, automatically update the image tag in repo 'infra' (e.g., in a values.yaml or deploy config), commit, and push."
Both repos must be linked in Repositories tab
Use signal.repo to identify which repo triggered
Clone repo B with its own URL (hardcoded or from secrets)
Need write access token (GH_TOKEN) scoped to both repos
Filter by signal.repo == "backend" to avoid double-trigger
Answer
signals:
  - push

validation: true

vars:
  REF:          "@<< signal.push.ref >>"
  TRIGGER_REPO: "@<< signal.repo >>"
  SHA:          "@<< signal.push.sha >>"
  SOURCE:       "@<< signal.source >>"
  OWNER:        "@<< signal.owner >>"
  TOKEN:        "@<< secrets.GH_TOKEN >>"

steps:
  - id: guard
    exec: |
      # Only act when backend pushes to main
      if [ "$TRIGGER_REPO" != "backend" ] || \
         [ "$REF" != "refs/heads/main" ]; then
        echo "Not a backend/main push. Skipping."
        exit 0
      fi

  - id: update-infra
    exec: |
      set -e
      INFRA_URL="https://$TOKEN@$SOURCE.com/$OWNER/infra"

      git clone "$INFRA_URL" infra
      cd infra

      # Replace image tag in values.yaml
      sed -i "s|tag:.*|tag: \"$SHA\"|g" helm/values.yaml

      git config user.email "ci@pulse-runner"
      git config user.name "Pulse Runner CI"
      git add helm/values.yaml
      git commit -m "chore: update backend image to $SHA"
      git push origin main
Tip: Link the processor to BOTH repos in the Repositories tab so it receives events from both. The guard step ensures it only acts on the right one.

Auto-rollback on Deploy Failure
push expert
Likely prompt
"Deploy on push to main. If the deploy step fails, automatically revert the repository to the previous commit and push the revert. Log both outcomes."
Save previous SHA (HEAD^) before deploy
Use set +e to capture deploy exit code without aborting
On failure: git revert HEAD --no-edit and push
Write outcome to OUTPUT_PATH for validation expression
push main → checkout → deploy (set +e) ├─ success → log ok → validation=true └─ failure → git revert + push → validation=true
Answer
signals:
  - push

validation: true

vars:
  REF:      "@<< signal.push.ref >>"
  PUSH_SHA: "@<< signal.push.sha >>"
  REPO_URL: "https://@<< secrets.GH_TOKEN >>@@<< signal.source >>.com/@<< signal.owner >>/@<< signal.repo >>"

steps:
  - id: checkout
    exec: |
      set -e
      [ "$REF" != "refs/heads/main" ] && exit 0
      git clone "$REPO_URL" .
      git fetch origin "$PUSH_SHA" && git checkout "$PUSH_SHA"

  - id: deploy-with-rollback
    exec: |
      set +e   # don't abort on error
      echo "Deploying $PUSH_SHA..."

      # --- your real deploy command here ---
      ./deploy.sh
      DEPLOY_EXIT=$?

      set -e
      if [ $DEPLOY_EXIT -eq 0 ]; then
        echo "Deploy succeeded ✓"
        echo "result=deployed" >> "$OUTPUT_PATH"
      else
        echo "Deploy FAILED. Rolling back..."
        git config user.email "ci@pulse-runner"
        git config user.name "Pulse Runner CI"
        git revert HEAD --no-edit
        git push origin main
        echo "result=rolled-back" >> "$OUTPUT_PATH"
        echo "Rollback pushed. Last good SHA: $(git rev-parse HEAD)"
      fi
⚠ The rollback push will fire another push signal. Make sure your deploy step is idempotent or add a commit message guard (like in #17) to avoid rollback loops.

Security Scan Gate — Block PR on Vulnerability
pull_request hard security
Likely prompt
"On every pull request, run a dependency vulnerability scan (e.g., npm audit, trivy, or govulncheck). If HIGH or CRITICAL vulnerabilities are found, fail the run and output a summary of findings."
Use npm audit --audit-level=high (non-zero exit = vulns found)
Or use trivy fs --exit-code 1 --severity HIGH,CRITICAL .
Capture output with tee to both log and file
Write scan_passed to OUTPUT_PATH for dynamic validation
Answer
signals:
  - pull_request

validation: true

vars:
  PR_ACTION: "@<< signal.pull_request.action >>"
  HEAD_SHA:  "@<< signal.pull_request.head_sha >>"
  REPO_URL:  "https://@<< secrets.GH_TOKEN >>@@<< signal.source >>.com/@<< signal.owner >>/@<< signal.repo >>"

steps:
  - id: guard
    exec: |
      [ "$PR_ACTION" = "opened" ] || \
      [ "$PR_ACTION" = "synchronize" ] || exit 0

  - id: checkout
    exec: |
      set -e
      git clone --depth 1 "$REPO_URL" .
      git fetch origin "$HEAD_SHA" && git checkout "$HEAD_SHA"

  - id: cache-deps
    cache:
      key: [package-lock.json]
      paths: [node_modules/]

  - id: security-scan
    exec: |
      npm ci --ignore-scripts
      set +e
      npm audit --audit-level=high 2>&1 | tee audit.txt
      AUDIT_EXIT=$?
      set -e

      if [ $AUDIT_EXIT -ne 0 ]; then
        echo "=== SECURITY SCAN FAILED ==="
        cat audit.txt
        echo "scan_passed=false" >> "$OUTPUT_PATH"
        exit 1
      else
        echo "=== No HIGH/CRITICAL vulnerabilities found ==="
        echo "scan_passed=true" >> "$OUTPUT_PATH"
      fi

PR Auto-labeler by Changed Path
pull_request medium
Likely prompt
"When a PR is opened, inspect the changed files. If changes are in src/frontend/, apply label 'frontend'. If in src/api/, apply label 'backend'. Use the GitHub API to set the label."
Use git diff between base_sha and head_sha
Call GitHub API: PATCH /repos/:owner/:repo/issues/:number/labels
Auth with secrets.GH_TOKEN as Bearer token
Use signal.pull_request.number as the issue number
Answer
signals:
  - pull_request

validation: true

vars:
  PR_ACTION: "@<< signal.pull_request.action >>"
  PR_NUM:    "@<< signal.pull_request.number >>"
  HEAD_SHA:  "@<< signal.pull_request.head_sha >>"
  BASE_SHA:  "@<< signal.pull_request.base_sha >>"
  OWNER:     "@<< signal.owner >>"
  REPO:      "@<< signal.repo >>"
  TOKEN:     "@<< secrets.GH_TOKEN >>"
  REPO_URL:  "https://@<< secrets.GH_TOKEN >>@@<< signal.source >>.com/@<< signal.owner >>/@<< signal.repo >>"

steps:
  - id: guard
    exec: |
      [ "$PR_ACTION" = "opened" ] || exit 0

  - id: checkout
    exec: |
      set -e
      git clone --depth 50 "$REPO_URL" .
      git fetch origin "$HEAD_SHA" "$BASE_SHA"
      git checkout "$HEAD_SHA"

  - id: label
    exec: |
      set -e
      CHANGED=$(git diff --name-only "$BASE_SHA" "$HEAD_SHA")
      LABELS="[]"

      echo "$CHANGED" | grep -q "^src/frontend/" && \
        LABELS=$(echo "$LABELS" | jq '. + ["frontend"]')
      echo "$CHANGED" | grep -q "^src/api/" && \
        LABELS=$(echo "$LABELS" | jq '. + ["backend"]')

      if [ "$LABELS" = "[]" ]; then
        echo "No matching paths found."
        exit 0
      fi

      # Apply labels via GitHub API
      curl -sf -X POST \
        -H "Authorization: Bearer $TOKEN" \
        -H "Content-Type: application/json" \
        -d "{\"labels\": $LABELS}" \
        "https://api.github.com/repos/$OWNER/$REPO/issues/$PR_NUM/labels"
      echo "Applied labels: $LABELS"

Scheduled Stale Issue Closer
schedule hard
Likely prompt
"Every day at midnight, find all open issues that haven't been updated in 30+ days and close them with a comment explaining they were closed for inactivity. Use the GitHub API."
Schedule cron: "0 0 * * *" (midnight daily)
GitHub API: GET /repos/:owner/:repo/issues?state=open&per_page=100
Calculate age with date -d arithmetic
POST comment then PATCH issue state to closed
Answer
signals:
  - schedule

schedule:
  cron: "0 0 * * *"
  source: github
  owner: Pulse-Runner
  repo: my-repo

validation: true

vars:
  OWNER: "@<< signal.owner >>"
  REPO:  "@<< signal.repo >>"
  TOKEN: "@<< secrets.GH_TOKEN >>"

steps:
  - id: close-stale
    exec: |
      set -e
      STALE_DAYS=30
      NOW=$(date +%s)

      # Fetch open issues
      curl -sf \
        -H "Authorization: Bearer $TOKEN" \
        "https://api.github.com/repos/$OWNER/$REPO/issues?state=open&per_page=100" \
        > issues.json

      CLOSED=0
      for NUMBER in $(jq -r '.[].number' issues.json); do
        UPDATED=$(jq -r --argjson n $NUMBER \
          '.[] | select(.number==$n) | .updated_at' issues.json)
        UPDATED_TS=$(date -d "$UPDATED" +%s)
        AGE_DAYS=$(( (NOW - UPDATED_TS) / 86400 ))

        if [ "$AGE_DAYS" -ge "$STALE_DAYS" ]; then
          echo "Closing stale issue #$NUMBER (age: ${AGE_DAYS}d)"

          # Post closing comment
          curl -sf -X POST \
            -H "Authorization: Bearer $TOKEN" \
            -H "Content-Type: application/json" \
            -d '{"body":"Closing due to 30 days of inactivity."}' \
            "https://api.github.com/repos/$OWNER/$REPO/issues/$NUMBER/comments"

          # Close the issue
          curl -sf -X PATCH \
            -H "Authorization: Bearer $TOKEN" \
            -H "Content-Type: application/json" \
            -d '{"state":"closed"}' \
            "https://api.github.com/repos/$OWNER/$REPO/issues/$NUMBER"

          CLOSED=$((CLOSED + 1))
        fi
      done
      echo "Closed $CLOSED stale issue(s)."

Performance Benchmark & Regression Gate
pull_request expert perf
Likely prompt
"On every PR, build both base and head commits and run a benchmark. If the PR introduces a performance regression greater than 10%, fail the run and output the comparison."
Checkout base_sha → run bench → save result
Checkout head_sha → run bench → compare
Use shell arithmetic (awk) for percentage comparison
Cache build deps to keep bench results unaffected by setup time
Answer
signals:
  - pull_request

validation: true

vars:
  PR_ACTION: "@<< signal.pull_request.action >>"
  HEAD_SHA:  "@<< signal.pull_request.head_sha >>"
  BASE_SHA:  "@<< signal.pull_request.base_sha >>"
  REPO_URL:  "https://@<< secrets.GH_TOKEN >>@@<< signal.source >>.com/@<< signal.owner >>/@<< signal.repo >>"

steps:
  - id: guard
    exec: |
      [ "$PR_ACTION" = "opened" ] || \
      [ "$PR_ACTION" = "synchronize" ] || exit 0

  - id: bench-base
    exec: |
      set -e
      git clone --depth 100 "$REPO_URL" base_repo
      cd base_repo
      git fetch origin "$BASE_SHA" && git checkout "$BASE_SHA"
      go build ./...
      # Run benchmark, extract ns/op
      go test -bench=BenchmarkMain -benchtime=3s ./... 2>&1 \
        | grep "BenchmarkMain" \
        | awk '{print $3}' \
        | sed 's/ns\/op//' > /tmp/base_ns.txt
      echo "base_ns=$(cat /tmp/base_ns.txt)" >> "$OUTPUT_PATH"

  - id: bench-head
    exec: |
      set -e
      git clone --depth 50 "$REPO_URL" head_repo
      cd head_repo
      git fetch origin "$HEAD_SHA" && git checkout "$HEAD_SHA"
      go build ./...
      go test -bench=BenchmarkMain -benchtime=3s ./... 2>&1 \
        | grep "BenchmarkMain" \
        | awk '{print $3}' \
        | sed 's/ns\/op//' > /tmp/head_ns.txt
      echo "head_ns=$(cat /tmp/head_ns.txt)" >> "$OUTPUT_PATH"

  - id: compare
    exec: |
      BASE_NS="@<< steps.bench-base.base_ns >>"
      HEAD_NS="@<< steps.bench-head.head_ns >>"

      echo "Base: ${BASE_NS} ns/op"
      echo "Head: ${HEAD_NS} ns/op"

      REGRESSION=$(awk "BEGIN {
        diff = ($HEAD_NS - $BASE_NS) / $BASE_NS * 100
        printf \"%.1f\", diff
      }")
      echo "Regression: ${REGRESSION}%"

      FAIL=$(awk "BEGIN { print ($REGRESSION > 10) ? 1 : 0 }")
      if [ "$FAIL" = "1" ]; then
        echo "FAIL: performance regression ${REGRESSION}% > 10% threshold"
        exit 1
      fi
      echo "PASS: regression within acceptable range"

Mirror Repo Between Platforms (GitHub → Gitea)
push multi-platform hard
Likely prompt
"Whenever a push happens on GitHub, mirror the entire repository (all branches and tags) to the corresponding Gitea repository. Use credentials for both platforms."
Clone from GitHub with secrets.GH_TOKEN
Add Gitea as a remote with secrets.GITEA_TOKEN
git push --mirror gitea syncs all refs at once
Gitea URL format: https://<token>@<gitea-host>/<owner>/<repo>
Answer
signals:
  - push

validation: true

vars:
  OWNER:        "@<< signal.owner >>"
  REPO:         "@<< signal.repo >>"
  GH_TOKEN:     "@<< secrets.GH_TOKEN >>"
  GITEA_TOKEN:  "@<< secrets.GITEA_TOKEN >>"
  GITEA_HOST:   gitea.pulse-runner.csgames.org

steps:
  - id: mirror
    exec: |
      set -e
      GH_URL="https://$GH_TOKEN@github.com/$OWNER/$REPO.git"
      GITEA_URL="https://$GITEA_TOKEN@$GITEA_HOST/$OWNER/$REPO.git"

      # Clone bare (gets all branches + tags)
      git clone --bare "$GH_URL" mirror.git
      cd mirror.git

      # Push everything to Gitea
      git remote add gitea "$GITEA_URL"
      git push --mirror gitea
      echo "Mirror complete: $OWNER/$REPO → Gitea"
--bare + --mirror is the cleanest way — copies all refs (branches, tags, notes) without a working tree. Never use a regular clone for mirroring.

Dynamic Validation Gate with Multiple Conditions
pull_request expert
Likely prompt
"A PR should only count as validated if: (1) it targets main, (2) the PR title follows conventional commits, (3) tests pass, AND (4) no secrets are hardcoded. All four must pass. Output a detailed report."
Run all checks independently, collect results
AND all four pass flags into a single all_passed output

    exec: |
      [ "$PR_ACTION" = "opened" ] || \

  - id: checkout

      set -e
      git clone --depth 50 "$REPO_URL" .

    exec: |
      PASS_TARGET=false; PASS_TITLE=false
      PASS_TESTS=false;  PASS_SECRETS=false

      # 1. Targets main
      [ "$BASE_REF" = "refs/heads/main" ] && PASS_TARGET=true
      echo "  target-main:   $PASS_TARGET"

      # 2. Conventional commit title
      PATTERN="^(feat|fix|chore|docs|refactor|test|ci)(\(.+\))?: .+"
      echo "$PR_TITLE" | grep -qE "$PATTERN" && PASS_TITLE=true
      echo "  title-format:  $PASS_TITLE  ($PR_TITLE)"

      # 3. Tests
      set +e
      npm ci --silent && npm test --silent 2>&1
      [ $? -eq 0 ] && PASS_TESTS=true
      set -e
      echo "  tests:         $PASS_TESTS"

      # 4. No hardcoded secrets (look for common patterns)
      set +e
      git grep -rE \
        "(AKIA[0-9A-Z]{16}|ghp_[a-zA-Z0-9]{36}|glpat-[a-zA-Z0-9\-]{20})" \
        -- '*.js' '*.ts' '*.py' '*.go' '*.yaml' '*.json'
      [ $? -ne 0 ] && PASS_SECRETS=true
      set -e
      echo "  no-secrets:    $PASS_SECRETS"

      # AND all together
      ALL=false
      [ "$PASS_TARGET"  = "true" ] && \
      [ "$PASS_TITLE"   = "true" ] && \
      [ "$PASS_TESTS"   = "true" ] && \
      [ "$PASS_SECRETS" = "true" ] && ALL=true

      echo "---"
      echo "all-passed: $ALL"
      echo "all_passed=$ALL" >> "$OUTPUT_PATH"

validation: "@<< steps.gate.all_passed >>"

Maximum-Score Optimized Push Pipeline
push send expert max score
Likely prompt
"Design a pipeline that maximizes leaderboard score: minimize CPU time, memory, disk I/O, and network. Build, test, and produce an artifact — but do it as efficiently as possible."
--depth 1 clone always — never full history
Cache everything: deps, tools, even Docker layer cache
Skip steps when nothing changed (check-changes pattern from #15)
npm ci over npm install — faster and reproducible
Stream build output to artifact, never copy large files unnecessarily
Use --ignore-scripts where possible to skip postinstall overhead
push → cache(node_modules) restore if hit → shallow clone + checkout → diff check → skip if no src changes → npm ci → skipped if cache hit → build → output path to artifact → artifact dist/ only, no node_modules
Answer — maximum efficiency
signals:
  - push

validation: true

vars:
  REPO_URL: "https://@<< secrets.GH_TOKEN >>@@<< signal.source >>.com/@<< signal.owner >>/@<< signal.repo >>"
  PUSH_SHA: "@<< signal.push.sha >>"
  REF:      "@<< signal.push.ref >>"

steps:
  # STEP 1: Restore dep cache BEFORE clone
  # (cache is keyed off the lock file from last run)
  - id: restore-deps
    cache:
      key: [package-lock.json]
      paths: [node_modules/]

  # STEP 2: Shallow clone only — no full history
  - id: checkout
    exec: |
      set -e
      git clone --depth 2 "$REPO_URL" .
      git fetch origin "$PUSH_SHA"
      git checkout "$PUSH_SHA"

  # STEP 3: Skip build if nothing relevant changed
  - id: diff-check
    exec: |
      CHANGED=$(git diff --name-only HEAD^ HEAD 2>/dev/null || echo "src/")
      if echo "$CHANGED" | grep -qE '^(src/|package\.json|package-lock\.json)'; then
        echo "needs_build=true" >> "$OUTPUT_PATH"
      else
        echo "No src changes, skipping build."
        echo "needs_build=false" >> "$OUTPUT_PATH"
      fi

  # STEP 4: Install only if needed, prefer cached
  - id: install
    exec: |
      NEEDS="@<< steps.diff-check.needs_build >>"
      [ "$NEEDS" = "true" ] || exit 0
      npm ci --ignore-scripts --prefer-offline 2>/dev/null || npm ci

  # STEP 5: Build — write exact output path to avoid unnecessary copies
  - id: build
    exec: |
      NEEDS="@<< steps.diff-check.needs_build >>"
      [ "$NEEDS" = "true" ] || exit 0
      set -e
      npm run build -- --outDir=dist
      echo "out=dist" >> "$OUTPUT_PATH"

  # STEP 6: Update cache with potentially new lock file
  - id: save-deps
    cache:
      key: [package-lock.json]
      paths: [node_modules/]

# Only include dist — never node_modules in artifact
artifacts:
  - name: dist
    paths:
      - "@<< steps.build.out >>"
Score tips: Cache hit = ~0 network in for deps. --depth 2 (not 1) lets you diff HEAD^ HEAD for change detection. --prefer-offline avoids registry calls entirely when cache is warm. Never put node_modules in artifacts — it's huge and kills your disk + network scores.

02
Clone & Checkout Commit
push easy
Likely prompt
"On every push to the repository, clone it and check out the exact pushed commit. Use the repository secret for authentication."
Use secrets.GH_TOKEN in clone URL
Use signal.push.sha for exact commit
Use signal.source / owner / repo for URL
Answer
signals:
  - push

validation: true

vars:
  REPO_URL: "https://@<< secrets.GH_TOKEN >>@@<< signal.source >>.com/@<< signal.owner >>/@<< signal.repo >>"
  PUSH_SHA: "@<< signal.push.sha >>"

steps:
  - id: checkout
    exec: |
      git clone --depth 1 "$REPO_URL" .
      git fetch origin "$PUSH_SHA"
      git checkout "$PUSH_SHA"
Tip: --depth 1 saves bandwidth/time. git fetch is needed when the sha isn't the tip of the cloned branch.

03
Run Tests on Pull Request
pull_request medium
Likely prompt
"When a pull request is opened or updated (synchronized), clone the head commit and run the test suite (e.g., npm test or go test ./...)."
Filter to opened or synchronize actions only
Checkout head_sha not push sha
Exit non-zero to mark run as failed if tests fail
Answer
signals:
  - pull_request

validation: true

vars:
  REPO_URL: "https://@<< secrets.GH_TOKEN >>@@<< signal.source >>.com/@<< signal.owner >>/@<< signal.repo >>"
  HEAD_SHA: "@<< signal.pull_request.head_sha >>"
  PR_ACTION: "@<< signal.pull_request.action >>"

steps:
  - id: guard
    exec: |
      # Only run on open/sync, skip everything else
      if [ "$PR_ACTION" != "opened" ] && \
         [ "$PR_ACTION" != "synchronize" ]; then
        echo "Skipping action: $PR_ACTION"
        exit 0
      fi

  - id: checkout
    exec: |
      set -e
      git clone --depth 1 "$REPO_URL" .
      git fetch origin "$HEAD_SHA"
      git checkout "$HEAD_SHA"

  - id: test
    exec: |
      set -e
      npm ci
      npm test
      # or: go test ./...
      # or: python -m pytest
04
Filter by Branch (main only)
push easy
Likely prompt
"Run a build only when code is pushed to the main branch. Ignore pushes to all other branches."
signal.push.ref = "refs/heads/main"
Compare ref string in bash, exit 0 if wrong branch
Don't set validation: true on skipped runs — use dynamic validation
Answer
signals:
  - push

vars:
  REF: "@<< signal.push.ref >>"
  REPO_URL: "https://@<< secrets.GH_TOKEN >>@@<< signal.source >>.com/@<< signal.owner >>/@<< signal.repo >>"
  PUSH_SHA: "@<< signal.push.sha >>"

steps:
  - id: check-branch
    exec: |
      if [ "$REF" != "refs/heads/main" ]; then
        echo "Not main, skipping."
        echo "valid=false" >> "$OUTPUT_PATH"
        exit 0
      fi
      echo "valid=true" >> "$OUTPUT_PATH"

  - id: build
    exec: |
      set -e
      git clone --depth 1 "$REPO_URL" .
      git fetch origin "$PUSH_SHA"
      git checkout "$PUSH_SHA"
      npm ci && npm run build

validation: "@<< steps.check-branch.valid >>"

05
Docker Build & Push on PR
pull_request hard
Likely prompt
"When a pull request is opened or synchronized, build a Docker image tagged with the head commit SHA, push it to a registry, and notify a deploy processor with the image tag."
Tag image as name:head_sha
Use secrets.DOCKER_USER and secrets.DOCKER_TOKEN
Write image_tag to OUTPUT_PATH for later steps
Use sends to chain to deploy processor
Answer
signals:
  - pull_request

validation: true

vars:
  IMAGE_NAME: myapp
  PR_ACTION:  "@<< signal.pull_request.action >>"
  REPO_URL:   "https://@<< secrets.GH_TOKEN >>@@<< signal.source >>.com/@<< signal.owner >>/@<< signal.repo >>"

steps:
  - id: checkout
    exec: |
      set -e
      if [ "$PR_ACTION" != "opened" ] && \
         [ "$PR_ACTION" != "synchronize" ]; then
        exit 0
      fi
      git clone --depth 1 "$REPO_URL" .
      git fetch origin "$HEAD_SHA"
      git checkout "$HEAD_SHA"
    vars:
      HEAD_SHA: "@<< signal.pull_request.head_sha >>"

  - id: build-image
    exec: |
      set -e
      TAG="$IMAGE_NAME:$HEAD_SHA"
      docker build -t "$TAG" .
      echo "image_tag=$TAG" >> "$OUTPUT_PATH"
    vars:
      HEAD_SHA: "@<< signal.pull_request.head_sha >>"

  - id: push-image
    exec: |
      set -e
      echo "$DOCKER_TOKEN" | docker login -u "$DOCKER_USER" --password-stdin
      docker push "$IMAGE_TAG"
    vars:
      DOCKER_USER:  "@<< secrets.DOCKER_USER >>"
      DOCKER_TOKEN: "@<< secrets.DOCKER_TOKEN >>"
      IMAGE_TAG:    "@<< steps.build-image.image_tag >>"

sends:
  - name: deploy-processor
    inputs:
      image: "@<< steps.build-image.image_tag >>"
      pr:    "@<< signal.pull_request.number >>"

Performance & Caching

06
Cache Dependencies (npm / Go / pip)
push medium
Likely prompt
"Build the project on push. Use caching to avoid re-downloading dependencies on every run. The cache should invalidate when the lock file changes."
Cache step before install step
Key on lock file (package-lock.json / go.sum / requirements.txt)
Huge bonus points — saves network + CPU
Answer (Node.js example — adapt for Go/Python)
signals:
  - push

validation: true

vars:
  REPO_URL: "https://@<< secrets.GH_TOKEN >>@@<< signal.source >>.com/@<< signal.owner >>/@<< signal.repo >>"
  PUSH_SHA: "@<< signal.push.sha >>"

steps:
  - id: checkout
    exec: |
      set -e
      git clone --depth 1 "$REPO_URL" .
      git fetch origin "$PUSH_SHA"
      git checkout "$PUSH_SHA"

  - id: cache-deps   # restore or create cache
    cache:
      key:
        - package-lock.json  # go.sum / requirements.txt
      paths:
        - node_modules/      # /root/go/pkg/mod / .venv

  - id: install
    exec: npm ci   # go mod download / pip install -r requirements.txt

  - id: build
    exec: npm run build
Go: key: go.sum · paths: /root/go/pkg/mod
Python: key: requirements.txt · paths: .venv/
Always put cache step before install — platform auto-skips install if cache hits.

Comments & Issues

08
Comment-Triggered Workflow (/deploy)
comment medium
Likely prompt
"When a user posts a comment containing '/deploy' on a pull request or issue, trigger a deployment. Ignore all other comments and ignore edited/deleted comment actions."
Check action == created first
grep comment body for /deploy trigger word
Use dynamic validation to avoid false positives
Answer
signals:
  - comment

vars:
  ACTION: "@<< signal.comment.action >>"
  BODY:   "@<< signal.comment.body >>"

steps:
  - id: check
    exec: |
      set -e
      if [ "$ACTION" != "created" ]; then
        echo "Not a new comment, skipping."
        echo "valid=false" >> "$OUTPUT_PATH"
        exit 0
      fi
      if ! echo "$BODY" | grep -q "/deploy"; then
        echo "No /deploy trigger found."
        echo "valid=false" >> "$OUTPUT_PATH"
        exit 0
      fi
      echo "valid=true" >> "$OUTPUT_PATH"

  - id: deploy
    exec: |
      set -e
      echo "Deploying as requested by comment..."
      # your deploy script here

validation: "@<< steps.check.valid >>"
⚠ If validation is always true, editing/deleting comments will also trigger runs and possibly confuse the validation script.
09
Auto-create Branch from Issue
issue hard
Likely prompt
"When a new issue is opened, automatically create a Git branch named 'issue-<number>' in the repository and push it."
Filter to action == opened
Use signal.issue.number to name branch
Push branch with GH_TOKEN auth
Answer
signals:
  - issue

validation: true

vars:
  ACTION:       "@<< signal.issue.action >>"
  ISSUE_NUM:    "@<< signal.issue.number >>"
  REPO_URL:     "https://@<< secrets.GH_TOKEN >>@@<< signal.source >>.com/@<< signal.owner >>/@<< signal.repo >>"

steps:
  - id: guard
    exec: |
      if [ "$ACTION" != "opened" ]; then
        echo "Skipping: $ACTION"
        exit 0
      fi

  - id: create-branch
    exec: |
      set -e
      BRANCH="issue-$ISSUE_NUM"
      git clone --depth 1 "$REPO_URL" repo
      cd repo
      git checkout -b "$BRANCH"
      git push origin "$BRANCH"
      echo "branch=$BRANCH" >> "$OUTPUT_PATH"

Sends & Chains

10
Chain Processors via Send
push send hard
Likely prompt
"Build a two-stage pipeline: a 'build' processor runs on push, compiles the code, and sends the binary path to a 'publish' processor which packages and archives it."
Build processor uses sends to trigger publish
Publish processor declares signal: send
Access inputs via signal.send.inputs.<key>
Processor 1: build-processor
signals:
  - push

vars:
  REPO_URL: "https://@<< secrets.GH_TOKEN >>@@<< signal.source >>.com/@<< signal.owner >>/@<< signal.repo >>"
  PUSH_SHA: "@<< signal.push.sha >>"

steps:
  - id: checkout
    exec: |
      set -e
      git clone --depth 1 "$REPO_URL" .
      git fetch origin "$PUSH_SHA" && git checkout "$PUSH_SHA"
  - id: compile
    exec: |
      set -e
      go build -o ./bin/app ./...
      echo "binary=./bin/app" >> "$OUTPUT_PATH"

sends:
  - name: publish-processor
    inputs:
      binary: "@<< steps.compile.binary >>"
      sha:    "@<< signal.push.sha >>"
Processor 2: publish-processor
signals:
  - send

validation: true

vars:
  BINARY: "@<< signal.send.inputs.binary >>"
  SHA:    "@<< signal.send.inputs.sha >>"

steps:
  - id: package
    exec: |
      set -e
      tar czf "app-$SHA.tar.gz" "$BINARY"
      echo "archive=app-$SHA.tar.gz" >> "$OUTPUT_PATH"

artifacts:
  - name: release
    paths:
      - "@<< steps.package.archive >>"

Validation Tricks

12
Validate PR Title Format
pull_request medium
Likely prompt
"Enforce that every PR title starts with a conventional commit prefix (feat:, fix:, chore:, docs:). Fail the run if it doesn't. Only check opened/synchronize PRs."
Use signal.pull_request.title
grep with regex for prefix check
exit 1 on failure → run marked failed
Answer
signals:
  - pull_request

validation: true

vars:
  PR_TITLE:  "@<< signal.pull_request.title >>"
  PR_ACTION: "@<< signal.pull_request.action >>"

steps:
  - id: lint-title
    exec: |
      set -e
      if [ "$PR_ACTION" != "opened" ] && \
         [ "$PR_ACTION" != "synchronize" ]; then
        exit 0
      fi
      PATTERN="^(feat|fix|chore|docs|refactor|test|ci|style)(\(.+\))?: .+"
      if echo "$PR_TITLE" | grep -qE "$PATTERN"; then
        echo "✓ Title is valid: $PR_TITLE"
      else
        echo "✗ Bad title format: '$PR_TITLE'"
        echo "  Must start with feat:, fix:, chore:, etc."
        exit 1
      fi
13
Deploy Only on PR Merge to Main
pull_request hard
Likely prompt
"Run a deployment script only when a pull request is closed AND it was merged into main (not just abandoned). Skip all other PR events."
action == closed does NOT mean merged
Check base_ref == refs/heads/main
No direct "merged" field — infer from action=closed + base=main
Answer
signals:
  - pull_request

vars:
  PR_ACTION:  "@<< signal.pull_request.action >>"
  BASE_REF:   "@<< signal.pull_request.base_ref >>"
  HEAD_SHA:   "@<< signal.pull_request.head_sha >>"
  REPO_URL:   "https://@<< secrets.GH_TOKEN >>@@<< signal.source >>.com/@<< signal.owner >>/@<< signal.repo >>"

steps:
  - id: guard
    exec: |
      if [ "$PR_ACTION" != "closed" ] || \
         [ "$BASE_REF" != "refs/heads/main" ]; then
        echo "Not a merge to main. Skipping."
        echo "valid=false" >> "$OUTPUT_PATH"
        exit 0
      fi
      echo "valid=true" >> "$OUTPUT_PATH"

  - id: deploy
    exec: |
      set -e
      git clone --depth 1 "$REPO_URL" .
      git fetch origin "$HEAD_SHA"
      git checkout "$HEAD_SHA"
      echo "Deploying $HEAD_SHA to production..."
      # ./deploy.sh or docker run etc.

validation: "@<< steps.guard.valid >>"

Full Multi-stage Pipeline (Build → Test → Deploy)
push send hard
Processor 1: ci-build (push trigger)
signals:
  - push
vars:
  REPO_URL: "https://@<< secrets.GH_TOKEN >>@
@<< signal.source >>.com/
@<< signal.owner >>/@<< signal.repo >>"
  SHA: "@<< signal.push.sha >>"
steps:
  - id: cache-deps
    cache:
      key: [package-lock.json]
      paths: [node_modules/]
  - id: checkout
    exec: |
      set -e
      git clone --depth 1 "$REPO_URL" .
      git fetch origin "$SHA"
      git checkout "$SHA"
  - id: build
    exec: |
      npm ci
      npm run build
      echo "dist=dist/" >> "$OUTPUT_PATH"
sends:
  - name: ci-test
    inputs:
      sha: "@<< signal.push.sha >>"
      dist: "@<< steps.build.dist >>"
Processor 2: ci-test (send trigger)
signals:
  - send
vars:
  SHA:  "@<< signal.send.inputs.sha >>"
  DIST: "@<< signal.send.inputs.dist >>"
  REPO_URL: "https://@<< secrets.GH_TOKEN >>@
@<< signal.source >>.com/
@<< signal.owner >>/@<< signal.repo >>"
steps:
  - id: cache-deps
    cache:
      key: [package-lock.json]
      paths: [node_modules/]
  - id: checkout
    exec: |
      git clone --depth 1 "$REPO_URL" .
      git fetch origin "$SHA"
      git checkout "$SHA"
  - id: test
    exec: |
      npm ci
      npm test
      echo "passed=true" >> "$OUTPUT_PATH"
sends:
  - name: ci-deploy
    inputs:
      sha:    "@<< signal.send.inputs.sha >>"
      passed: "@<< steps.test.passed >>"
Processor 3: ci-deploy (send trigger)
signals:
  - send

validation: true

vars:
  SHA:    "@<< signal.send.inputs.sha >>"
  PASSED: "@<< signal.send.inputs.passed >>"
  REPO_URL: "https://@<< secrets.GH_TOKEN >>@
@<< signal.source >>.com/
@<< signal.owner >>/@<< signal.repo >>"
steps:
  - id: guard
    exec: |
      if [ "$PASSED" != "true" ]; then
        echo "Tests failed, skip deploy."
        exit 1
      fi
  - id: deploy
    exec: |
      set -e
      git clone --depth 1 "$REPO_URL" .
      git fetch origin "$SHA"
      git checkout "$SHA"
      echo "Deploying $SHA..."
      # ./deploy.sh or kubectl apply etc.
Remember: All 3 processors must be linked to the same repository in the Repositories tab.

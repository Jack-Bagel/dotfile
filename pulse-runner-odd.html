Actor Allowlist — Ignore Bots & Outsiders
push quirky easy
Likely prompt
"Only run the build when the push was made by a human team member. Ignore pushes from bots (actors ending in [bot]) and unknown contributors not in the approved list."
Use signal.actor to get the username
grep actor against a hardcoded allowlist or [bot] suffix check
Bots often have names like dependabot[bot], github-actions[bot]
Answer
signals:
  - push

vars:
  ACTOR: "@<< signal.actor >>"

steps:
  - id: actor-check
    exec: |
      # Reject bots
      if echo "$ACTOR" | grep -q '\[bot\]'; then
        echo "Actor is a bot ($ACTOR). Skipping."
        echo "allowed=false" >> "$OUTPUT_PATH"
        exit 0
      fi

      # Check against team allowlist
      ALLOWED="alice bob charlie diana"
      if echo "$ALLOWED" | grep -qw "$ACTOR"; then
        echo "Actor $ACTOR is approved."
        echo "allowed=true" >> "$OUTPUT_PATH"
      else
        echo "Actor $ACTOR not in allowlist. Skipping."
        echo "allowed=false" >> "$OUTPUT_PATH"
      fi

  - id: build
    exec: |
      [ "@<< steps.actor-check.allowed >>" = "true" ] || exit 0
      echo "Running build for $ACTOR..."

validation: "@<< steps.actor-check.allowed >>"
Note: This also prevents bot-triggered pushes (like from #17's version bump) from re-triggering downstream pipelines.

Commit Message Command Parser
push quirky medium
Likely prompt
"Parse the commit message for special instructions embedded as [skip-tests], [notify], or [force-deploy]. Each tag changes processor behavior: skip the test suite, send a notification, or bypass the branch guard."
Read last commit message with git log -1 --pretty=%B
Use grep to check for each tag and write flags to OUTPUT_PATH
Downstream steps read those flags to decide what to do
Answer
signals:
  - push

validation: true

vars:
  REPO_URL: "https://@<< secrets.GH_TOKEN >>@@<< signal.source >>.com/@<< signal.owner >>/@<< signal.repo >>"
  SHA:      "@<< signal.push.sha >>"

steps:
  - id: checkout
    exec: |
      set -e
      git clone --depth 1 "$REPO_URL" .
      git fetch origin "$SHA" && git checkout "$SHA"

  - id: parse-msg
    exec: |
      MSG=$(git log -1 --pretty=%B)
      echo "Commit message: $MSG"

      echo "$MSG" | grep -q '\[skip-tests\]'   && SKIP_TESTS=true  || SKIP_TESTS=false
      echo "$MSG" | grep -q '\[notify\]'        && NOTIFY=true      || NOTIFY=false
      echo "$MSG" | grep -q '\[force-deploy\]'  && FORCE=true       || FORCE=false

      echo "skip_tests=$SKIP_TESTS"   >> "$OUTPUT_PATH"
      echo "notify=$NOTIFY"           >> "$OUTPUT_PATH"
      echo "force_deploy=$FORCE"      >> "$OUTPUT_PATH"

  - id: test
    exec: |
      if [ "@<< steps.parse-msg.skip_tests >>" = "true" ]; then
        echo "[skip-tests] detected — skipping test suite."
        exit 0
      fi
      npm ci && npm test

  - id: notify
    exec: |
      if [ "@<< steps.parse-msg.notify >>" = "true" ]; then
        echo "Sending notification for $SHA..."
        curl -sf -X POST "$WEBHOOK_URL" \
          -d "{\"text\": \"Build triggered: $SHA\"}"
      fi
    vars:
      WEBHOOK_URL: "@<< secrets.WEBHOOK_URL >>"

  - id: deploy
    exec: |
      REF="@<< signal.push.ref >>"
      FORCE="@<< steps.parse-msg.force_deploy >>"
      if [ "$REF" != "refs/heads/main" ] && [ "$FORCE" != "true" ]; then
        echo "Not main and no [force-deploy]. Skipping deploy."
        exit 0
      fi
      echo "Deploying..."

PR Description Checklist Enforcer
pull_request quirky medium
Likely prompt
"Every PR must have a description body containing a checklist with at least these three boxes checked: '- [x] Tests added', '- [x] Docs updated', '- [x] Reviewed'. Fail if any are unchecked."
Use signal.pull_request.body to read PR description
Checked box = - [x], unchecked = - [ ]
grep for exact checked patterns
Body can be empty on lazy PRs — handle that null case
Answer
signals:
  - pull_request

validation: true

vars:
  PR_ACTION: "@<< signal.pull_request.action >>"
  PR_BODY:   "@<< signal.pull_request.body >>"

steps:
  - id: checklist
    exec: |
      [ "$PR_ACTION" = "opened" ] || \
      [ "$PR_ACTION" = "edited" ] || exit 0

      if [ -z "$PR_BODY" ]; then
        echo "PR body is empty. All checks fail."
        exit 1
      fi

      FAIL=0
      check() {
        LABEL="$1"; PATTERN="$2"
        if echo "$PR_BODY" | grep -qi "$PATTERN"; then
          echo "  ✓ $LABEL"
        else
          echo "  ✗ $LABEL — not checked"
          FAIL=1
        fi
      }

      check "Tests added"   '\- \[x\] Tests added'
      check "Docs updated"  '\- \[x\] Docs updated'
      check "Reviewed"      '\- \[x\] Reviewed'

      if [ $FAIL -ne 0 ]; then
        echo "PR checklist incomplete. Fix the PR description."
        exit 1
      fi
      echo "All checklist items confirmed ✓"
Tip: Use action: edited too — authors often fix the body after opening. Without it they'd need to close and re-open to re-trigger.

Issue Title → Webhook Notification
issue quirky easy
Likely prompt
"When an issue tagged '[URGENT]' is opened, immediately send a webhook POST (e.g. to Slack or a custom endpoint) containing the issue number, title, and author."
Use signal.issue.title to detect the [URGENT] tag
Build JSON payload with jq to safely escape values
Store webhook URL in secrets.WEBHOOK_URL
Only fire on action: opened
Answer
signals:
  - issue

validation: true

vars:
  ACTION:      "@<< signal.issue.action >>"
  TITLE:       "@<< signal.issue.title >>"
  NUMBER:      "@<< signal.issue.number >>"
  ACTOR:       "@<< signal.actor >>"
  REPO:        "@<< signal.repo >>"
  WEBHOOK_URL: "@<< secrets.WEBHOOK_URL >>"

steps:
  - id: notify
    exec: |
      [ "$ACTION" = "opened" ] || exit 0

      if ! echo "$TITLE" | grep -qi '\[URGENT\]'; then
        echo "Not urgent. Skipping."
        exit 0
      fi

      PAYLOAD=$(jq -n \
        --arg repo   "$REPO"   \
        --arg num    "$NUMBER" \
        --arg title  "$TITLE"  \
        --arg author "$ACTOR"  \
        '{
          text: ("🚨 URGENT issue #" + $num + " in " + $repo),
          issue: { number: $num, title: $title, author: $author }
        }')

      curl -sf -X POST "$WEBHOOK_URL" \
        -H "Content-Type: application/json" \
        -d "$PAYLOAD"
      echo "Webhook sent for issue #$NUMBER"
jq -n builds JSON from scratch safely — avoids injection from special chars in titles like quotes or backslashes.

Run Exactly Once — Idempotent Bootstrap
send quirky medium
Likely prompt
"Create a processor that sets up a tool or initializes a resource exactly once. If triggered again, it should detect the prior setup and skip — even if sent many times."
Write a sentinel file or git tag as a "done" marker
Check for its existence at the start of the run
Useful for one-time repo scaffolding, DB migrations, etc.
Can use a git tag like initialized to check state
Answer (uses a git tag as sentinel)
signals:
  - send

validation: true

vars:
  REPO_URL: "https://@<< secrets.GH_TOKEN >>@@<< signal.source >>.com/@<< signal.owner >>/@<< signal.repo >>"
  TOKEN:    "@<< secrets.GH_TOKEN >>"
  OWNER:    "@<< signal.owner >>"
  REPO:     "@<< signal.repo >>"

steps:
  - id: check-sentinel
    exec: |
      # Check if sentinel tag exists via API (no clone needed)
      STATUS=$(curl -sf -o /dev/null -w "%{http_code}" \
        -H "Authorization: Bearer $TOKEN" \
        "https://api.github.com/repos/$OWNER/$REPO/git/refs/tags/initialized")

      if [ "$STATUS" = "200" ]; then
        echo "Already initialized. Nothing to do."
        echo "did_run=false" >> "$OUTPUT_PATH"
        exit 0
      fi
      echo "did_run=true" >> "$OUTPUT_PATH"

  - id: setup
    exec: |
      [ "@<< steps.check-sentinel.did_run >>" = "true" ] || exit 0
      set -e
      git clone --depth 1 "$REPO_URL" .
      git config user.email "ci@pulse-runner"
      git config user.name  "Pulse Runner CI"

      # Do your one-time setup here
      echo "# Initialized by CI" >> README.md
      git add README.md
      git commit -m "chore: CI bootstrap"

      # Push sentinel tag to mark as done
      git tag initialized
      git push origin main --tags
      echo "Bootstrap complete."

Comment Reaction Counter — Close on 👍 Threshold
comment quirky medium
Likely prompt
"When a comment containing '+1' or '👍' is posted on an issue, count the total number of such comments. If the count reaches 3, automatically close the issue as resolved."
Trigger on comment signal, filter action: created
Use GitHub API to list all comments on the issue
Count how many contain +1 or 👍 in body
PATCH issue to closed when threshold met
Answer
signals:
  - comment

validation: true

vars:
  ACTION:  "@<< signal.comment.action >>"
  BODY:    "@<< signal.comment.body >>"
  NUMBER:  "@<< signal.comment.number >>"
  OWNER:   "@<< signal.owner >>"
  REPO:    "@<< signal.repo >>"
  TOKEN:   "@<< secrets.GH_TOKEN >>"
  THRESHOLD: 3

steps:
  - id: count-upvotes
    exec: |
      [ "$ACTION" = "created" ] || exit 0

      # Get all comments for this issue
      curl -sf \
        -H "Authorization: Bearer $TOKEN" \
        "https://api.github.com/repos/$OWNER/$REPO/issues/$NUMBER/comments?per_page=100" \
        > comments.json

      # Count +1 / 👍 comments
      COUNT=$(jq '[.[] | select(.body | test("\\+1|👍"))] | length' comments.json)
      echo "Upvote count: $COUNT / $THRESHOLD"
      echo "count=$COUNT" >> "$OUTPUT_PATH"

      if [ "$COUNT" -ge "$THRESHOLD" ]; then
        echo "Threshold reached — closing issue #$NUMBER"
        curl -sf -X PATCH \
          -H "Authorization: Bearer $TOKEN" \
          -H "Content-Type: application/json" \
          -d '{"state":"closed","state_reason":"completed"}' \
          "https://api.github.com/repos/$OWNER/$REPO/issues/$NUMBER"

        curl -sf -X POST \
          -H "Authorization: Bearer $TOKEN" \
          -H "Content-Type: application/json" \
          -d "{\"body\": \"Closing automatically — reached $THRESHOLD upvotes ✅\"}" \
          "https://api.github.com/repos/$OWNER/$REPO/issues/$NUMBER/comments"
      fi

Time-of-Day Gate — No Deploys on Friday
push quirky easy
Likely prompt
"Block all production deploys on Fridays after 3pm UTC and on weekends. If a push comes in during those windows, skip the deploy and log a warning instead."
Use bash date +%u for day of week (1=Mon, 5=Fri, 6=Sat, 7=Sun)
Use date +%H for UTC hour
No signal field for this — purely runtime date check
Classic "no Friday deploys" rule
Answer
signals:
  - push

validation: true

vars:
  REF: "@<< signal.push.ref >>"

steps:
  - id: time-gate
    exec: |
      [ "$REF" = "refs/heads/main" ] || exit 0

      DOW=$(date -u +%u)   # 1=Mon ... 5=Fri, 6=Sat, 7=Sun
      HOUR=$(date -u +%H)  # 00-23 UTC

      echo "Day: $DOW  Hour: $HOUR UTC"

      BLOCKED=false
      [ "$DOW" -ge 6 ] && BLOCKED=true   # weekend
      [ "$DOW" -eq 5 ] && [ "$HOUR" -ge 15 ] && BLOCKED=true  # Fri after 15:00

      if [ "$BLOCKED" = "true" ]; then
        echo "⛔ Deploy blocked — no Friday-evening/weekend deploys."
        echo "deploy=false" >> "$OUTPUT_PATH"
        exit 0
      fi

      echo "✅ Time check passed. Proceeding with deploy."
      echo "deploy=true" >> "$OUTPUT_PATH"

  - id: deploy
    exec: |
      [ "@<< steps.time-gate.deploy >>" = "true" ] || exit 0
      echo "Deploying..."
Real world: This is a genuine best practice. More incidents happen from Friday afternoon deploys than any other time.

Cron Health Watchdog — Alert if Processor Goes Silent
schedule quirky medium
Likely prompt
"Run every hour. Fetch the timestamp of the last successful run of a specific processor via the platform or a sentinel file. If it hasn't run in 2+ hours, send a webhook alert."
Schedule acts as a watchdog heartbeat
The monitored processor writes a timestamp to a repo file on each run
Watchdog reads that file, compares to current time
Alert if delta exceeds threshold
Answer
signals:
  - schedule

schedule:
  cron: "0 * * * *"  # every hour
  source: github
  owner: Pulse-Runner
  repo: ops-repo

validation: true

vars:
  REPO_URL:    "https://@<< secrets.GH_TOKEN >>@github.com/Pulse-Runner/ops-repo"
  WEBHOOK_URL: "@<< secrets.WEBHOOK_URL >>"
  MAX_AGE_SEC: 7200  # 2 hours

steps:
  - id: watchdog
    exec: |
      set -e
      git clone --depth 1 "$REPO_URL" .

      # Monitored processor writes: echo "$(date -u +%s)" > .last_run
      if [ ! -f .last_run ]; then
        echo "No .last_run file found — first run or never ran."
        exit 0
      fi

      LAST=$(cat .last_run)
      NOW=$(date -u +%s)
      DELTA=$((NOW - LAST))
      echo "Last run: ${DELTA}s ago (max: ${MAX_AGE_SEC}s)"

      if [ "$DELTA" -gt "$MAX_AGE_SEC" ]; then
        echo "⚠ Processor has been silent for ${DELTA}s!"
        curl -sf -X POST "$WEBHOOK_URL" \
          -H "Content-Type: application/json" \
          -d "{\"text\": \"⚠ Watchdog: processor silent for ${DELTA}s (threshold: ${MAX_AGE_SEC}s)\"}"
      else
        echo "✅ Processor is healthy."
      fi
Pair it: The monitored processor should update .last_run on every success: echo "$(date -u +%s)" > .last_run && git add .last_run && git commit -m "heartbeat" && git push

Count Lines of Code & Fail if Over Budget
pull_request quirky easy
Likely prompt
"On every PR, count the total lines added (not deleted). If a single PR adds more than 500 lines of code, fail the run with a message asking the author to split it up."
Use git diff --stat or --numstat for line counts
Diff between base_sha and head_sha
--numstat gives added/deleted per file — sum the added column
Answer
signals:
  - pull_request

validation: true

vars:
  PR_ACTION: "@<< signal.pull_request.action >>"
  HEAD_SHA:  "@<< signal.pull_request.head_sha >>"
  BASE_SHA:  "@<< signal.pull_request.base_sha >>"
  REPO_URL:  "https://@<< secrets.GH_TOKEN >>@@<< signal.source >>.com/@<< signal.owner >>/@<< signal.repo >>"
  MAX_LINES: 500

steps:
  - id: guard
    exec: |
      [ "$PR_ACTION" = "opened" ] || \
      [ "$PR_ACTION" = "synchronize" ] || exit 0

  - id: checkout
    exec: |
      set -e
      git clone --depth 50 "$REPO_URL" .
      git fetch origin "$HEAD_SHA" "$BASE_SHA"

  - id: count-lines
    exec: |
      set -e
      # --numstat: added  deleted  filename
      ADDED=$(git diff --numstat "$BASE_SHA" "$HEAD_SHA" \
        | awk '{sum += $1} END {print sum}')

      echo "Lines added in this PR: $ADDED"

      if [ "$ADDED" -gt "$MAX_LINES" ]; then
        echo "❌ PR adds $ADDED lines — exceeds budget of $MAX_LINES."
        echo "Please split this PR into smaller pieces."
        exit 1
      fi
      echo "✅ Line count ($ADDED) is within budget ($MAX_LINES)."

File Size Guard — Block Large Binary Commits
push quirky easy
Likely prompt
"When a push is made, scan all changed files. If any single file exceeds 5MB, or if the total size of changed files exceeds 20MB, fail the run and list the offenders."
Use git diff --name-only HEAD^ HEAD to list changed files
Check each file with du -b or wc -c
Accumulate total size with awk
Common issue: accidentally committed node_modules or build artifacts
Answer
signals:
  - push

validation: true

vars:
  REPO_URL:     "https://@<< secrets.GH_TOKEN >>@@<< signal.source >>.com/@<< signal.owner >>/@<< signal.repo >>"
  SHA:          "@<< signal.push.sha >>"
  MAX_FILE_MB:  5
  MAX_TOTAL_MB: 20

steps:
  - id: checkout
    exec: |
      set -e
      git clone --depth 2 "$REPO_URL" .
      git fetch origin "$SHA" && git checkout "$SHA"

  - id: size-check
    exec: |
      set -e
      MAX_FILE_BYTES=$((MAX_FILE_MB * 1024 * 1024))
      MAX_TOTAL_BYTES=$((MAX_TOTAL_MB * 1024 * 1024))
      FAIL=0
      TOTAL=0

      CHANGED=$(git diff --name-only HEAD^ HEAD 2>/dev/null || git diff --name-only HEAD)

      for FILE in $CHANGED; do
        [ -f "$FILE" ] || continue
        SIZE=$(wc -c < "$FILE")
        TOTAL=$((TOTAL + SIZE))
        SIZE_MB=$(awk "BEGIN {printf \"%.2f\", $SIZE/1048576}")
        if [ "$SIZE" -gt "$MAX_FILE_BYTES" ]; then
          echo "❌  $FILE — ${SIZE_MB}MB exceeds ${MAX_FILE_MB}MB limit"
          FAIL=1
        else
          echo "✅  $FILE — ${SIZE_MB}MB"
        fi
      done

      TOTAL_MB=$(awk "BEGIN {printf \"%.2f\", $TOTAL/1048576}")
      echo "---"
      echo "Total changed: ${TOTAL_MB}MB"

      if [ "$TOTAL" -gt "$MAX_TOTAL_BYTES" ]; then
        echo "❌ Total size ${TOTAL_MB}MB exceeds ${MAX_TOTAL_MB}MB budget"
        FAIL=1
      fi

      [ $FAIL -eq 0 ] || exit 1
      echo "All file sizes within limits ✅"

push quirky medium

"Store your processor YAML in the repository at .ci/processor.yaml. On every push, read that file, and use it as the processor config — effectively making the processor definition version-controlled and self-updating."

signals:

Random Reviewer Assignment on PR Open
pull_request quirky medium
Likely prompt
"When a PR is opened, randomly assign one reviewer from the team roster (excluding the PR author). Use the GitHub API to set the reviewer."
Filter reviewer list to exclude signal.actor
Use $RANDOM % count for random selection in bash
GitHub API: POST /repos/:owner/:repo/pulls/:number/requested_reviewers
Only trigger on action: opened
Answer
signals:
  - pull_request

validation: true

vars:
  PR_ACTION: "@<< signal.pull_request.action >>"
  PR_NUM:    "@<< signal.pull_request.number >>"
  ACTOR:     "@<< signal.actor >>"
  OWNER:     "@<< signal.owner >>"
  REPO:      "@<< signal.repo >>"
  TOKEN:     "@<< secrets.GH_TOKEN >>"

steps:
  - id: assign-reviewer
    exec: |
      [ "$PR_ACTION" = "opened" ] || exit 0

      # Team roster — exclude the PR author
      ALL_REVIEWERS="alice bob charlie diana eve"
      CANDIDATES=""
      for R in $ALL_REVIEWERS; do
        [ "$R" != "$ACTOR" ] && CANDIDATES="$CANDIDATES $R"
      done

      # Pick one randomly
      CANDIDATES_ARR=($CANDIDATES)
      COUNT=${#CANDIDATES_ARR[@]}
      PICK=${CANDIDATES_ARR[$((RANDOM % COUNT))]}
      echo "Assigning reviewer: $PICK"

      curl -sf -X POST \
        -H "Authorization: Bearer $TOKEN" \
        -H "Content-Type: application/json" \
        -d "{\"reviewers\": [\"$PICK\"]}" \
        "https://api.github.com/repos/$OWNER/$REPO/pulls/$PR_NUM/requested_reviewers"
      echo "Reviewer $PICK assigned to PR #$PR_NUM"

Emoji-only PR Title Gate 🚀
pull_request quirky easy
Likely prompt
"Your team decided every PR title must start with an emoji (because why not). Fail the run if the title doesn't begin with a Unicode emoji character. Bonus: extract and log which emoji was used."
Emoji characters are multi-byte Unicode — use Python or grep with Unicode range
Python unicodedata can detect emoji category
Alternatively: check if title starts with a known emoji via a regex
Genuinely useful for visual categorization in PR lists
Answer
signals:
  - pull_request

validation: true

vars:
  PR_ACTION: "@<< signal.pull_request.action >>"
  PR_TITLE:  "@<< signal.pull_request.title >>"

steps:
  - id: emoji-check
    exec: |
      [ "$PR_ACTION" = "opened" ] || \
      [ "$PR_ACTION" = "edited" ] || exit 0

      # Use Python to detect leading emoji
      python3 - <<'PYEOF'
import sys, os, unicodedata

title = os.environ.get("PR_TITLE", "")
if not title:
    print("Empty title")
    sys.exit(1)

first_char = title[0]
cat = unicodedata.category(first_char)
name = unicodedata.name(first_char, "unknown")

# Emoji are typically So (Symbol, other) or have high code points
is_emoji = ord(first_char) > 127 and cat in ("So", "Sm", "Sk", "Ll") \
           or ord(first_char) > 0x1F300

if is_emoji:
    print(f"✅ PR title starts with emoji: {first_char} ({name})")
    sys.exit(0)
else:
    print(f"❌ PR title must start with an emoji. Got: '{first_char}'")
    sys.exit(1)
PYEOF
Fun teams use this to quickly scan PRs: 🐛 = bugfix, ✨ = feature, 🔥 = delete code, 🚀 = deploy, 📝 = docs.

Detect & Block Force Pushes
push quirky medium
Likely prompt
"Detect when a push to main is a force push (history was rewritten). If detected, alert via webhook and optionally create an issue documenting the incident with timestamp and actor."
A force push means the pushed SHA is not a descendant of the previous tip
Compare signal.push.sha against git merge-base with previous HEAD
Or: use GitHub API to get the ref's previous SHA before the push
Simpler heuristic: if git log --oneline origin/main..$SHA is empty but SHA != old tip
Answer
signals:
  - push

validation: true

vars:
  REF:     "@<< signal.push.ref >>"
  SHA:     "@<< signal.push.sha >>"
  ACTOR:   "@<< signal.actor >>"
  OWNER:   "@<< signal.owner >>"
  REPO:    "@<< signal.repo >>"
  TOKEN:   "@<< secrets.GH_TOKEN >>"
  REPO_URL: "https://@<< secrets.GH_TOKEN >>@@<< signal.source >>.com/@<< signal.owner >>/@<< signal.repo >>"
  WEBHOOK: "@<< secrets.WEBHOOK_URL >>"

steps:
  - id: detect-force-push
    exec: |
      [ "$REF" = "refs/heads/main" ] || exit 0

      # Clone with slightly more history to detect rewrite
      git clone --depth 10 "$REPO_URL" .
      git fetch origin "$SHA"

      # Get previous remote tip before this push
      PREV_TIP=$(git rev-parse origin/main 2>/dev/null || echo "")

      if [ -z "$PREV_TIP" ] || [ "$PREV_TIP" = "$SHA" ]; then
        echo "First push or same SHA. Not a force push."
        exit 0
      fi

      # If pushed SHA is not a descendant of previous tip → force push
      if git merge-base --is-ancestor "$PREV_TIP" "$SHA" 2>/dev/null; then
        echo "✅ Normal push — $SHA descends from $PREV_TIP"
        exit 0
      fi

      echo "🚨 FORCE PUSH DETECTED on main by $ACTOR"
      echo "  Previous tip: $PREV_TIP"
      echo "  New tip:      $SHA"

      # Alert via webhook
      curl -sf -X POST "$WEBHOOK" \
        -H "Content-Type: application/json" \
        -d "{\"text\":\"🚨 Force push to main by $ACTOR ($SHA)\"}"

      # Create incident issue
